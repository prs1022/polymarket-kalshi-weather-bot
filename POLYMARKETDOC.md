## 1. System Architecture Overview

Polymarket operates as a hybrid decentralized exchange (DEX) combining off-chain order matching with on-chain settlement.

*   **Settlement Layer**: Polygon (POS) Network.
*   **Settlement Asset**: `pUSD` (Polymarket USD) - upgraded from legacy USDC.e.
*   **Market Mechanism**: Conditional Tokens Framework (ERC-1155) developed by Gnosis.
*   **Order Matching Engine**: Central Limit Order Book (CLOB).

### Core Components
1.  **Gamma API**: The content management and market discovery engine.
2.  **CLOB API**: The high-frequency, non-custodial trading engine.
3.  **Data API**: The historical analytics and indexing ledger.

## 3. Core Protocol Mechanics (Critical for AI Logic)

### 3.1 Conditional Tokens (ERC-1155)
Markets are represented by `Condition` IDs. Each condition resolves to a vector of probabilities.
*   **Binary Markets**: Two outcomes (YES / NO). ERC-1155 tokens are minted dynamically.
*   **Categorical Markets**: Multiple outcomes (e.g., Candidate A, Candidate B, Candidate C).
*   **Token Pricing**: Prices always fluctuate between `$0.00` and `$1.00`. The sum of all outcomes in a standard market equals `$1.00`.

### 3.2 Negative Risk Mechanism (`negRisk`)
For multi-outcome categorical markets (3+ choices), Polymarket uses a **Negative Risk** architecture.
*   **Implementation**: Traders bet against outcomes by buying the "NO" equivalent across grouped conditions.
*   **AI Code Requirement**: When interacting with Multi-Outcome endpoints or contracts, the flag `negRisk: true` must be included in order payloads, otherwise transactions will revert.

### 3.3 The Heartbeat & Cancel on Disconnect
To protect market makers from API network disconnects, Polymarket implements a server-side **Dead Man's Switch**.
*   **Frequency**: API Clients must send a heartbeat every `10 seconds`.
*   **Grace Period**: `5 seconds`.
*   **Behavior**: If no heartbeat is received within 15 seconds, the CLOB engine automatically cancels all open orders for that API key.

## 4. Authentication & Onboarding Flow

To interact with the CLOB API for trading, you must derive an API key pair using an Ethereum private key registered on Polymarket.

### Step 1: Derive the Polymarket API Credentials
The credentials consist of an `API Key`, an `API Secret`, and an `API Passphrase`. These are generated by signing a specific EIP-712 message with your Polymarket Proxy Wallet private key.

### Step 2: Request Headers Structure
Every authenticated private request must contain the following HTTP headers:

```http
POLY-API-KEY: <your_api_key>
POLY-API-SIGN: <hmac_sha256_signature_of_payload>
POLY-API-TIMESTAMP: <current_unix_timestamp_seconds>
POLY-API-PASSPHRASE: <your_api_passphrase>
```

## 7. Operational Guardrails for Trading Bots

1.  **Rate Limiting**: CLOB API applies strict rate limiting per IP and API key. Bursting past 100 requests/sec will result in an HTTP `429 Too Many Requests` status code. Implement exponential backoff.
2.  **Gas Fees**: Trading on the CLOB does *not* consume gas from your wallet directly; transaction matching fees are natively internalized or sponsored depending on your API volume profile. Gas is only spent when depositing or withdrawing funds via the bridge.
3.  **Decimal Precision**: Token counts use `6 decimals` precision matching the underlying pUSD stability layer. Always sanitize floats to fixed string precision before formatting JSON RPC packages.
